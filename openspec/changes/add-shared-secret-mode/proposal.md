## Why

mkClientTooling's `secret` capability only supports `prompted` (operator enters interactively) and `generated` (auto-creates via openssl) modes. Both create a **new** per-client vars generator. There's no way to reference a secret that was already generated by the **server** role and shared via `share = true`. This blocks services like subcog where the client needs the server's DB password and JWT token.

## What Changes

- Add a third secret mode `shared` that references an existing shared vars generator by name instead of creating a new one
- Secret file paths are exposed as `secretPaths` (attrset keyed by secret name), passed to `cli.envVars` and `mcp.extraConfig` callbacks
- Support multiple secrets per client (a client may need both a DB password and a JWT token from different shared generators)

## Capabilities

### New Capabilities
- `shared-secret`: Secret mode that references an existing server-side shared vars generator by name, enabling clients to consume secrets created and shared by the server role

### Modified Capabilities

## Impact

- **lib/mkClientTooling.nix**: Core changes to secret handling — new mode branch, multi-secret support, vars generator lookup for shared mode
- **Downstream services**: subcog (and any future service needing shared secrets) can wire server-generated secrets through to client CLI/MCP configs
- **No breaking changes**: Existing `prompted` and `generated` modes are unaffected; `secret` option shape expands but remains backward-compatible
