## Context

mkClientTooling generates a complete clanService client role from a capabilities declaration. Its `secret` capability currently supports two modes:

- **prompted**: Creates a vars generator with a hidden prompt; operator enters the secret interactively during deployment.
- **generated**: Creates a vars generator that runs `openssl rand -hex 32` with `share = true`.

Both modes create a **new** per-client vars generator keyed as `agentplot-${serviceName}-${clientName}-${secret.name}`. The resulting file path is currently named `tokenPath` internally and passed to `cli.envVars` and hardcoded into `mcpConfig.tokenFile`.

The problem: there's no way to reference a secret that was already generated by the **server** role. Services like subcog need the client to consume multiple server-side shared secrets (DB password, JWT token).

## Goals / Non-Goals

**Goals:**
- Add a `shared` secret mode that resolves `secretPath` to an existing shared vars generator's file path
- Support multiple secrets per client (e.g., both a DB password and a JWT token)
- Maintain backward compatibility with existing `prompted` and `generated` modes

**Non-Goals:**
- Changing how `prompted` or `generated` modes work
- Adding server-side secret generation logic (that already exists)
- Supporting cross-service secret sharing (only same-service server→client)

## Decisions

### D1: Shared mode resolves to an existing generator path, creates no new generator

In `shared` mode, `secretPath` will point to the file path of a vars generator that already exists (created by the server role). No new vars generator is created.

```nix
secret = {
  name = "db-password";
  mode = "shared";
  generator = "subcog-db-password";   # existing generator name
  file = "password";                  # file within that generator
};
```

The secretPath resolves to:
```nix
config.clan.core.vars.generators.${secret.generator}.files.${secret.file}.path
```

**Rationale**: The server role already creates the generator with `share = true`. The client just needs to reference it. Creating another generator would duplicate the secret.

**Alternative considered**: A separate `sharedSecrets` capability alongside `secret`. Rejected because it would duplicate the secretPath→envVars→mcpConfig plumbing that already exists.

### D2: Multi-secret via list, not single attrset

Change `capabilities.secret` from a single attrset to a list of secret attrsets. Each secret produces its own named path in a `secretPaths` attrset passed to `cli.envVars`.

```nix
capabilities.secret = [
  { name = "db-password"; mode = "shared"; generator = "subcog-db-password"; file = "password"; }
  { name = "jwt-token"; mode = "shared"; generator = "subcog-jwt-token"; file = "token"; }
];
```

The `cli.envVars` callback receives `secretPaths` (attrset keyed by secret name):
```nix
envVars = settings: {
  DB_PASSWORD_FILE = settings.secretPaths."db-password";
  JWT_TOKEN_FILE = settings.secretPaths."jwt-token";
};
```

**Rationale**: subcog needs two different secrets. A list is the minimal expansion that supports this without inventing a parallel mechanism.

**Alternative considered**: An attrset of secrets keyed by name. A list was chosen for consistency with other list-based capabilities like `skills`.

### D3: Backward-compatible secret normalization

Accept both the old single-attrset form and the new list form:

```nix
# Old form (still works)
secret = { name = "api-key"; mode = "prompted"; };

# New form
secret = [
  { name = "api-key"; mode = "prompted"; }
  { name = "db-pass"; mode = "shared"; generator = "foo-db-pass"; file = "password"; }
];
```

Early in `mkClientTooling`, normalize to list:
```nix
secrets = if builtins.isList secret then secret
          else if secret != null then [ secret ]
          else [ ];
hasSecret = secrets != [ ];
```

**Rationale**: No breaking change for existing services. Single-secret services don't need to change anything.

### D4: Replace `mcp.tokenFile` with generic `mcp.extraConfig` callback

The current `mcp.tokenFile` field is HTTP-auth-specific and only supports a single secret. MCP servers may need multiple secrets mapped to arbitrary config fields (e.g., `tokenFile`, `passwordFile`, `apiKeyFile`).

Replace the hardcoded `tokenFile` logic with a generic `mcp.extraConfig` callback — the same pattern `cli.envVars` already uses:

```nix
mcp = {
  type = "http";
  urlTemplate = settings: "...";
  extraConfig = settings: {
    tokenFile = settings.secretPaths."jwt-token";
    passwordFile = settings.secretPaths."db-password";
  };
};
```

The MCP config builder becomes:
```nix
mcpConfig =
  { inherit url; }
  // lib.optionalAttrs (mcp.type == "http") { type = "http"; }
  // lib.optionalAttrs (mcp ? extraConfig) (mcp.extraConfig (clientSettings // { inherit secretPaths; }));
```

For backward compatibility, if `mcp.tokenFile` is present (old form), synthesize an `extraConfig` that maps `tokenFile` to the first secret's path:
```nix
extraConfig = settings: { tokenFile = builtins.head (builtins.attrValues settings.secretPaths); };
```

**Rationale**: Services know what MCP config fields they need — mkClientTooling shouldn't hardcode field names. This mirrors the `cli.envVars` pattern and scales to any number of secrets or config fields.

**Alternative considered**: Keeping `tokenFile` and adding more named fields. Rejected because it doesn't generalize — every new MCP auth pattern would require a new hardcoded field in mkClientTooling.

## Risks / Trade-offs

- **[Risk] Shared generator doesn't exist at eval time** → The vars generator reference will fail with a clear Nix error if the server role isn't also applied. This is acceptable — it's a configuration error. No special validation needed.
- **[Risk] Old `tokenPath` callers need updating** → Existing `cli.envVars` callbacks that reference `settings.tokenPath` must be updated to `settings.secretPaths.<name>`. Only internal service definitions use this, not end users, so the migration surface is small.
- **[Trade-off] No runtime validation of shared secret accessibility** → We trust that clan's `share = true` mechanism handles file distribution. If the shared file doesn't exist at runtime, the CLI wrapper or MCP server will fail with a missing-file error, which is diagnosable.
